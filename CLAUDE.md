# CLAUDE.md

## 対話ルール

**ユーザーとの対話はすべて日本語で行うこと。**

## 絵文字使用禁止

ドキュメントおよび設定ファイルに絵文字を使用しないでください。

### 置換ルール

**設定フラグ系（有効/無効を示すもの）** → `true` / `false`
- 例: `| audit-black-hacker | true |`
- 例: `**ペルソナ有効**: false`

**ステータス・結果表示系** → テキストラベル
- `[OK]`（完了・成功を示す場合）
- `[NG]`（失敗・エラーを示す場合）
- `[注意]`（警告を示す場合）

**重要度ラベル系** → 角括弧付きテキスト
- `[Critical]`
- `[Major]`
- `[Minor]`
- `[Suggestion]`

## プロジェクト概要

Claude Code プラグイン配布リポジトリ。dbz-workflow と piwf を配布する。

## プロジェクト固有情報の混入禁止

このリポジトリ内のファイルには、**プロジェクト固有の情報を記述してはいけません**。

### 禁止例

- 特定プロジェクトのリポジトリURL
- 特定プロジェクトの技術スタック（例: 「このプロジェクトはNext.jsを使用」）
- 特定プロジェクトのディレクトリ構成
- 特定組織・チーム名

### OK例

- 汎用的な使用方法（例: 「コードレビューを行うエージェント」）
- 一般的なベストプラクティス
- プロジェクト非依存の設定テンプレート

## Git 運用ルール

- main ブランチへの直接コミット・プッシュを禁止する
- 作業は必ずフィーチャーブランチで行う
- PR を作成し、レビュー後にマージする
- `git add -A` / `git add .` の使用を禁止し、変更ファイルを明示的に指定すること
  - 理由: untracked ファイルの誤コミット防止

## バージョン管理ルール

プラグインを更新した場合、該当プラグインの `.claude-plugin/plugin.json` の `version` を必ず更新すること。

### 対象

プラグイン配下のファイルを変更した場合:

- スキル（`skills/` 配下）の変更
- エージェント（`agents/` 配下）の変更
- プラグイン設定（`.claude-plugin/` 配下）の変更
- Hooks（`hooks/` 配下）の変更
- その他プラグインの動作に影響するファイル

### 対象外

- README.md の誤字修正など、プラグインの動作に影響しないドキュメント変更

### セマンティックバージョニング

`MAJOR.MINOR.PATCH` 形式を採用する（[公式仕様](https://semver.org/lang/ja/)）。

| 種類 | 更新タイミング | 例 |
|------|---------------|-----|
| MAJOR | 後方互換性を持たない変更 | 1.2.3 → 2.0.0 |
| MINOR | 機能追加（後方互換性を維持） | 1.2.3 → 1.3.0 |
| PATCH | バグ修正（後方互換性を維持） | 1.2.3 → 1.2.4 |

**バージョン更新時の注意**:
- MAJOR を上げた場合は MINOR と PATCH を 0 にリセット
- MINOR を上げた場合は PATCH を 0 にリセット

## 情報捏造禁止ルール

機能・仕様・設定について記述する際は、**必ず事実確認を行うこと**。

### 事実確認の方法（優先順位順）

1. 実際のコード・設定ファイルを確認
2. **公式ドキュメントを WebFetch/WebSearch で確認する**（必須）
3. 上記すべてを試みた上で確認できなかった場合のみ、ユーザーに報告する

### 禁止事項

- **確認なしに「ありそうな」仕様を記述すること**
- **「要確認」「手動対応」等のラベルで作業を終わらせること**

### 不確かな場合の対応

1. **まず自分で確認を試みる**（WebFetch/WebSearch/context7で公式ドキュメントを検索）
2. 確認を試みた結果、それでも不明な場合のみユーザーに報告する
3. 報告時は「〜を確認しましたが見つかりませんでした」と試行内容を含める
