# コンテキスト管理ガイド

長期タスクやセッション跨ぎの作業において、エージェントがコンテキストを効果的に管理するためのガイド。

> **参考**: [Anthropic - Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)、[Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)

## プログレスファイル運用

### 目的

プログレスファイル（`PROGRESS.md`）は、セッション間で作業状態を引き継ぐためのファイル。git 履歴と併用することで、新しいセッションでも迅速に現状を把握できる。

### ファイル形式

```markdown
## 現在のステータス
[作業中の機能・タスクの状態を1-2行で記述]

## 完了済み
- [完了した作業1]
- [完了した作業2]

## 次のステップ
- [次に着手すべき作業1]
- [次に着手すべき作業2]

## ブロッカー・注意事項
- [既知の問題や失敗したアプローチとその理由]
```

### 運用ルール

| ルール | 説明 |
|--------|------|
| 記録タイミング | 機能単位の完了時、セッション終了前、ブロッカー発生時 |
| コミット対象外 | プログレスファイルは `.gitignore` に追加するか、コミットに含めない |
| 簡潔さ | 各項目1-2行。詳細はコードやコミットメッセージに委ねる |
| 失敗の記録 | 失敗したアプローチは理由とともに記録する（再試行防止） |

### 失敗アプローチの記録例

```markdown
## ブロッカー・注意事項
- CSS Modules で動的スタイル切り替えを試みたが、SSR時にクラス名不一致が発生。
  インラインスタイルに切り替えて解決済み。
```

失敗の記録は、同じエージェント（または後続エージェント）が同じ失敗を繰り返すことを防ぐ。

## セッション初期化プロトコル

### 目的

セッション開始時に最小限のコストで現状を把握し、作業を再開する。

### 手順

```
1. pwd                              # 作業ディレクトリ確認
2. PROGRESS.md を Read              # 前回の状態を把握（存在する場合）
3. git log --oneline -10            # 最近の変更を確認
4. git diff --stat                  # 未コミットの変更を確認
5. startup-guide.md に従い起動      # 設定読み込み・Issue情報取得
6. 整合性確認                       # プログレスと実際の状態の一致を確認
```

### 初回セッション vs 継続セッション

| 状況 | 動作 |
|------|------|
| PROGRESS.md が存在しない | 初回セッション。Issue情報から直接作業開始 |
| PROGRESS.md が存在する | 継続セッション。プログレスの内容に基づき作業再開 |
| プログレスと実態が不一致 | 実際のコード状態を優先し、プログレスを更新 |

## 長期タスク時のコンテキストリセット戦略

### 目的

コンテキストウィンドウの肥大化を防ぎ、エージェントの判断精度を維持する。

### リセット判断基準

以下の状況でコンテキストリセットを検討する:

1. **Phase 境界での切り替え**: Phase 1 → 4A、Phase 4A → 4B の遷移時
2. **修正ループの蓄積**: 複数回の修正で古い差分やエラーログが蓄積した場合
3. **サブタスク完了時**: 大規模な機能実装で個別のサブタスクが完了した時点

### リセット時の引き継ぎ

リセット（fresh start）時は、親エージェントが以下の情報をプロンプトに含める:

- プログレスファイルの内容
- 変更ファイル一覧（`git diff --name-only`）
- 次に着手すべきタスク
- 前セッションでの重要な判断事項

> **注意**: `agent-resume-guide.md` の Phase 境界リセットルールも併せて参照すること。

### コンテキスト節約の原則

| 原則 | 実践 |
|------|------|
| ファイル再読み込みの最小化 | 一度読んだファイルの内容は要点を記憶し、再読み込みを避ける |
| エラーログの要約 | エラーの全文ではなく、要点（エラー種別・発生箇所・原因）のみ保持 |
| 完了タスクの解放 | 完了したサブタスクの詳細はプログレスファイルに移す |
| 差分の整理 | 古い差分はコミットしてコンテキストから除外する |

## agent-resume-guide.md との関係

本ガイドは `agent-resume-guide.md` を補完する:

| ガイド | スコープ |
|--------|----------|
| agent-resume-guide.md | エージェントの resume/fresh start の技術的な制御（agentId 管理、Phase 境界） |
| context-management-guide.md（本ガイド） | プログレスファイル運用、セッション初期化、コンテキスト節約の実践的な戦略 |

両ガイドは独立して参照可能だが、長期タスクでは併用することを推奨する。
