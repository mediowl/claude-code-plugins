# 計画ワークフロースキル

コーディング前に実装計画を作成・レビューするためのスキル。

## 使い方

```
/dbz-plan XXX          # Issue #XXXの計画を開始
/dbz-plan #XXX         # 同上（#プレフィックス付き）
```

## ワークフローフェーズ

```
Phase 1: 調査・分析
    ↓
Phase 2: ユーザーに確認（AskUserQuestion）
    ↓
Phase 3: 計画v0を作成 → Issueに投稿
    ↓
Phase 4: 計画レビュー（plan-reviewer）← 最大3ループ
    ↓
Phase 5: 承認 → /dbz-pr へ連携
```

## 使用エージェント

| エージェント | フェーズ | 用途 |
|-------------|---------|------|
| `plan-reviewer` | 4 | 実装計画のレビュー |

## 計画の構造

計画は以下の構造で GitHub Issue に投稿：

```markdown
## 実装計画 v{version}

### 概要
[1-2文のサマリー]

### 変更ファイル
- `path/to/file` - [変更内容]

### 実装ステップ
1. [ステップ]

### 影響範囲
- [影響]

### リスク・注意事項
- [リスク]

### テスト計画
- [ ] [テスト項目]
```

## レビュー観点（7項目）

plan-reviewer は以下の観点で計画を評価：

1. **アプローチの妥当性** - 解決策は適切か？
2. **影響範囲の網羅性** - 影響を受ける箇所が全て特定されているか？
3. **リスクの洗い出し** - 潜在的リスクがリストされているか？
4. **ステップの順序** - 実装順序は論理的か？
5. **依存関係の明確性** - ステップ間の依存関係は明確か？
6. **テスト計画の適切性** - テストは変更をカバーしているか？
7. **設計の一貫性** - プロジェクトのパターンに従っているか？

## /dbz-pr への連携

計画承認後、ユーザーは以下を選択可能：

1. **/dbz-pr を開始** - すぐに実装を開始
2. **計画を保留** - 後で実装
3. **計画を破棄** - キャンセル

## 必要環境

- Git リポジトリ
- GitHub CLI（`gh`）- Issue 操作用
