# Plan Workflow Skill

実装・改修の計画を検討するワークフロースキル。

## Input

- `issue_number`: Issue番号（例: `XXX`, `#XXX`）— 計画ワークフロー開始

## Usage Examples

```
/dbz-plan XXX          # Issue #XXXの計画を開始
/dbz-plan #XXX         # 同上（#付きでもOK）
```

---

## Workflow Phases

```
Phase 1: 調査・分析
    ↓
Phase 2-1: 不明点をユーザーに質問
    ↓
Phase 2-2: 複雑度判定（フェーズ分割検討）
    ↓
Phase 3: 計画v0作成 → Issueコメントに投稿
    ↓
Phase 4: plan-reviewerレビュー  ← 最大3回ループ
    ↓
Phase 5: 計画承認 → workflow-pr連携確認
```

> **Note**: `/dbz-plan`にはPhase 0（ブランチ準備）がありません。計画段階ではブランチ作成は不要なためです。実装時に`/dbz-pr`でブランチを作成します。

---

## Execution Flow

### 設定ファイル読み込み・Issue情報取得（ワークフロー開始時・必須）

ワークフロー開始前に以下を実行:

1. `.claude/dbz-workflow.config.md` を読み込む（なければフォールバック）
   - ペルソナ設定
   - プロジェクト情報
   - モデル設定
2. `gh issue view <number> --json title,body,comments` でIssue情報取得（本文+コメント）
3. 画像URLがあれば認証付きcurlでダウンロード
4. Task ツール呼び出し時、モデル設定に従い `model` パラメータを指定

**詳細仕様**: `../../docs/config-loader.md` および `../../docs/issue-context-guide.md` を参照

---

### サブエージェントのresume管理

> **agentId汚染防止**: 前回のスキル実行で取得したagentIdは使用しない。すべてのagentIdはこのスキル実行内で新規取得すること。異なるスキル実行間でagentIdを引き継ぐと、意図しないコンテキスト混入やコンテキストリミット到達の原因となる。

1. 1回目: 新規起動、agentIdを記録
2. 2回目以降: `resume` パラメータに agentId を指定

**詳細仕様**: `../../docs/agent-resume-guide.md` を参照

---

### ペルソナ読み込み（各フェーズ共通）

サブエージェント呼び出し前に実施:
1. ペルソナ有効チェック（`false` ならスキップ）
2. `.claude/dbz-workflow/{エージェント名}.persona.md` を試行
3. メイン設定のペルソナセクションを試行

**詳細仕様**: `../../docs/config-loader.md` を参照

---

### Phase 1: 調査・分析

1. `gh issue view <issue_number> --json title,body,comments` でIssue内容を取得（本文+コメント）
2. **Issueコメントの全件確認（必須）**: コメントが存在する場合、全てのコメントを読み、追加の要件・仕様変更・議論の結論を把握する。Issue本文だけで調査を開始しないこと
3. 関連コードの調査（Grep, Glob, Read）
4. 影響範囲の特定
5. 技術的な実現可能性の確認

> **重要**: Issue本文とコメントの両方を全て読んでから調査・分析を開始すること。コメントには本文にない追加要件や仕様の変更が含まれている可能性がある。

### Phase 2-1: 不明点をユーザーに質問

- 要件が曖昧な場合はAskUserQuestionで確認
- 複数のアプローチがある場合は選択肢を提示
- **勝手に決定しない** — 不明点は必ずユーザーに確認

### Phase 2-2: 複雑度判定（フェーズ分割検討）

Phase 2-1 で要件が明確になった後、作業規模を判定します。

#### 2段階判定フロー

```
[第1段階: 定量トリガー]
    ├─ 予想変更ファイル数 >= 5 → 第2段階へ
    ├─ 予想実装ステップ数 >= 8 → 第2段階へ
    └─ いずれも該当しない → Phase 3 へ（分割不要）

[第2段階: 定性確認]
    ├─ 独立した機能変更が複数存在する
    │   AND それぞれ独立してリリース可能
    │   → フェーズ分割を提案
    └─ 上記を満たさない → Phase 3 へ（分割不要）
```

#### 定量トリガー判定

Phase 1 の調査結果から以下を確認：

| トリガー | 閾値 | 判定基準 |
|----------|------|----------|
| 予想変更ファイル数 | >= 5 | 新規作成 + 既存ファイルの編集（テストファイル含む） |
| 予想実装ステップ数 | >= 8 | 計画書の「実装ステップ」に記載予定のステップ数 |

**カウント例**:
- API追加（2ファイル）+ テスト（1ファイル）+ UI（2ファイル）+ テスト（1ファイル）= 6ファイル → トリガー該当
- バグ修正（1ファイル）+ テスト修正（1ファイル）= 2ファイル → トリガー非該当

**いずれも該当しない場合**: 分割不要と判断し、Phase 3 へ進む。

#### 定性確認

定量トリガーに該当した場合、以下を確認：

1. **独立した機能変更が複数存在するか？**
   - 例: 「API追加」と「UI実装」は独立した機能変更
   - 例: 「バリデーション追加」と「エラーハンドリング」は同一機能の一部

2. **それぞれ独立してリリース可能か？**
   - 例: APIが先にマージされ、UIは後からでも問題ない → 独立可能
   - 例: UIがAPIに依存し、同時リリースが必要 → 独立不可

**両方を満たす場合**: フェーズ分割を提案する。

#### フェーズ分割の提案

分割が妥当と判断した場合、ユーザーに提案：

```
この作業は規模が大きいため、フェーズ分割を提案します。

## 分割案

### フェーズ1: [機能名]
- [変更内容1]
- [変更内容2]
予想変更ファイル: X件

### フェーズ2: [機能名]
- [変更内容3]
- [変更内容4]
予想変更ファイル: Y件

---
次のアクションを選択してください：
1. 分割を採用（子Issueを自動作成）
2. 分割せず続行（1つのPRで対応）
```

#### 子Issue自動作成

ユーザーが「分割を採用」を選択した場合：

1. 各フェーズに対応する子Issueを作成
   ```bash
   gh issue create --title "[親Issue名] - フェーズ1: [機能名]" --body "$(cat <<'EOF'
親Issue: #XXX

## 概要
[フェーズ1の内容]
EOF
)" --label "phase-split"
   ```

   > **Note**: `phase-split` ラベルは、フェーズ分割で作成された子Issueであることを示す。親Issueとの関連付けや、分割された作業の追跡に使用する。

2. 親Issueに分割情報をコメント
   ```bash
   gh issue comment XXX --body "$(cat <<'EOF'
フェーズ分割を実施しました。

- 子Issue #YYY: フェーズ1
- 子Issue #ZZZ: フェーズ2
EOF
)"
   ```

3. **現在のワークフローは最初のフェーズ（子Issue）で継続**
   - 親Issueではなく、最初の子Issueに対して計画を作成

#### 分割しない場合

「分割せず続行」を選択した場合、そのまま Phase 3 へ進む。

**計画書への記載**:

Phase 3 で作成する計画書に、フェーズ分割検討の結果を記載する：

```markdown
### フェーズ分割検討
- 予想変更ファイル数: X件
- 予想実装ステップ数: Y件
- 判定結果: 分割不要（1つのPRで対応）
- 理由: [例: 変更が密接に関連しており、独立リリースが困難なため]
```

この記載により、レビュワーが「なぜこの規模で1PRなのか」を把握できる。

### Phase 3: 計画v0作成

計画書をIssueコメントとして投稿：

```markdown
## 実装計画 v{version}

### 概要
[1-2文で要約]

### 変更ファイル
- `path/to/file1.ts` - [変更内容]
- `path/to/file2.ts` - [変更内容]

### 実装ステップ
1. [ステップ1]
2. [ステップ2]
3. [ステップ3]

### 影響範囲
- [影響1]
- [影響2]

### リスク・注意点
- [リスク1]
- [リスク2]

### テスト計画
- [ ] [テスト項目1]
- [ ] [テスト項目2]

---
[bot] Generated with Claude Code
```

### Phase 4: plan-reviewerレビュー

計画書を `plan-reviewer` でレビューする。

ペルソナ読み込みを実施（上記「ペルソナ読み込み（各フェーズ共通）」参照）

1. 計画書のMarkdownを準備
2. plan-reviewerエージェントでレビュー実行
   - 「以下の実装計画をレビューしてください」と依頼
   - 計画書全文を渡す
3. レビュー結果をIssueコメントに投稿
   ```bash
   gh issue comment XXX --body-file review-result.md
   ```

**レビュー観点**（詳細は plan-reviewer.md 参照）:
- アプローチ妥当性（プロジェクト設計思想との整合性）
- 影響範囲網羅性（ドキュメント更新漏れ等）
- リスク洗い出し（後方互換性・パフォーマンス・セキュリティ）
- ステップ順序・依存関係（論理的な実装順序）
- テスト計画適切性（変更内容のカバレッジ）
- 設計の一貫性（命名規則・ファイル配置・責務分離）

**見ない観点**（reviewer との責務分離）:
- コマンド構文の正確性 → implementer（実装時）
- grepパターンの網羅性 → implementer（実装時）
- コード品質 → reviewer（Phase 4A）

**ループ処理**:
- Critical/Majorがあれば計画を再構築（最大3回ループ）
- Minor/Suggestionのみなら承認へ進む
- ループごとにバージョン番号を増やす（v0 → v1 → v2 → v3）

### Phase 5: 計画承認

計画が承認されたら、ユーザーに確認：

```
計画が承認されました。

次のアクションを選択してください：
1. /dbz-prを開始（実装フェーズへ）
2. 計画を保留（後で実装）
3. 計画を破棄
```

**選択肢ごとのアクション**:

| 選択肢 | 実行内容 |
|--------|----------|
| 1. /dbz-pr開始 | `/dbz-pr <issue_number>` を実行し、実装フェーズへ移行（Issue番号を引き継ぐ） |
| 2. 計画を保留 | 計画はIssueコメントに残り、後日 `/dbz-pr` で再開可能 |
| 3. 計画を破棄 | Issueコメントに「計画破棄」を記録し、ワークフロー終了 |

**workflow-pr連携の詳細**:

1. **Issue番号の引き継ぎ**: `/dbz-plan`で使用したIssue番号をそのまま`/dbz-pr`に渡す
   ```
   /dbz-pr XXX  # Issue #XXXの実装を開始
   ```

2. **計画の参照**: `/dbz-pr`のimplementerはIssueコメントの中の**最新の計画**を参照して実装を行う

3. **データの保存場所**: 計画はすべてIssueコメントに保存されるため、別途データ引き継ぎは不要

## Absolute Rules

1. **不明点は勝手に決定しない** — 必ずユーザーに質問
   - 例: 「AとBどちらのアプローチが良いですか？」と選択肢を提示
2. **計画はIssueコメントに投稿** — 履歴として残す
   - 例: `gh issue comment <number> --body-file plan.md`
3. **Critical/Majorを放置しない** — 最大3回ループで修正
   - 例: plan-reviewerが「テスト計画が不十分」と指摘 → テスト項目を追加してv1を作成
4. **内部タスク番号に`#`使用禁止** — GitHub誤リンク防止
   - 例: [NG] `#1 調査` → ○ `1. 調査`
5. **バージョン管理** — v0から始まり、ループごとにv1, v2, v3と増やす
6. **フェーズ分割は提案のみ** — 分割実行はユーザー承諾後に行う
   - 例: 自動で子Issueを作成せず、必ず「分割を採用」の確認を取る
7. **Issueコメントを全て読んでから計画を開始する** — Issue本文だけで計画を開始しない
   - コメントには追加の要件・仕様変更・議論の結論が含まれている可能性がある
   - `gh issue view <number> --json title,body,comments` でコメントも必ず取得すること

## Error Handling

### Issueが見つからない場合

```
Error: Issue #XXX が見つかりません。

Issue番号を確認してください。
```

### 最大ループ回数到達

```
最大ループ回数（3回）に到達しました。

残りのCritical/Major:
- [issue list]

人間によるレビュー・判断が必要です。
```

**計画保存**: 最終バージョンの計画をIssueコメントに投稿し、「最大ループ到達・人間判断待ち」とラベル付けする。計画は破棄せず、人間が後から参照・修正できる状態で保存する。
