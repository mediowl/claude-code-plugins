# workflow-init

対話式で dbz-workflow 設定ファイルを生成するスキル。

## 使い方

```bash
/dbz-init
```

引数なしで実行すると、対話形式で以下の情報を収集します：

1. **プロジェクト情報**: プロジェクトタイプ、技術スタック、テストコマンド
2. **監査エージェント設定**: プロジェクトタイプに応じた推奨設定
3. **ペルソナ設定**: エージェントの口調・キャラクター（デフォルト / 共通 / エージェント別）
4. **カスタムルール**: プロジェクト固有のルール（デフォルト）

収集した情報を元に `.claude/dbz-workflow.config.md` を生成します。
ペルソナ設定で「共通ペルソナ」または「エージェント別ペルソナ」を選択した場合、`.claude/dbz-workflow/` ディレクトリにペルソナファイルも生成します。

---

## 対話例

### Phase 1: プロジェクト情報

**質問1:**
```
プロジェクトタイプは？
a) Webフロントエンド
b) バックエンドAPI
c) CLIツール
d) ライブラリ
e) ドキュメントのみ
```
**回答**: `e`

**質問2:**
```
テストコマンドは？（例: npm test）
a) npm test
b) npm run test
c) yarn test
d) pnpm test
e) その他（自由記述）
```
**回答**: `a`

**質問3:**
```
E2Eテストコマンドは？
a) npm run e2e
b) npm run test:e2e
c) npx playwright test
d) E2Eテストなし
e) その他（自由記述）
```
**回答**: `d`

**質問4:**
```
デザインシステムを使用していますか？（例: Material-UI, Chakra UI）
a) はい（使用している）
b) いいえ（使用していない）
```
**回答**: `b`

---

### Phase 2: 詳細設定

**質問1:**
```
多言語対応（i18n）を行っていますか？
a) はい（対応している）
b) いいえ（対応していない）
```
**回答**: `b`

**質問2:**
```
監査エージェント設定を確認しますか？
a) 推奨設定を使用（プロジェクトタイプに基づく）
b) カスタマイズする（個別に有効/無効を選択）
```
**回答**: `a`

---

### Phase 3: ペルソナ設定

**質問:**
```
ペルソナ設定はどうしますか？
a) デフォルト（丁寧な日本語）
b) 共通ペルソナ（全エージェント共通のペルソナを設定）
c) エージェント別ペルソナ（各エージェントに個別のペルソナを設定）
```
**回答**: `a`（デフォルトを選択した場合）

---

### ペルソナ設定の対話例（共通ペルソナ + 自動補完）

**Phase 3でbまたはcを選択した場合の例**

**質問1:**
```
{エージェント名}の名前は？（例: ベジータ、レビュー担当）
```
**回答**: `ベジータ`

**質問2:**
```
{エージェント名}の役割は？（例: コードレビューの王子、実装担当者）
```
**回答**: `コードレビューの王子`

**質問3:**
```
ベジータが登場する作品名は？（既製作品のキャラクターの場合のみ入力してください。オリジナルキャラの場合は空欄でEnter）
例: アニメ「ドラゴンボールZ」
```
**回答**: `アニメ「ドラゴンボールZ」`

**質問4:**
```
「アニメ『ドラゴンボールZ』」のキャラクター「ベジータ」の一人称・語尾・口調の特徴を、キャラクター性から自動補完しますか？
a) はい（自動補完する）
b) いいえ（手動で入力する）
```
**回答**: `a`

**確認:**
```
以下の内容で ベジータ のペルソナファイルを生成します。よろしいですか？

**一人称**: オレ
**語尾**: 〜だ、〜だな、〜だろう
**口調の特徴**:
- 断定的で自信に満ちた口調
- 指摘時は「甘いな」「話にならん」などの表現
- 承認時は「悪くない」「認めてやろう」などの表現

a) はい（この内容で生成）
b) いいえ（手動で入力し直す）
```
**回答**: `a`

### 生成される設定ファイル

**ペルソナ設定で「デフォルト」を選択した場合**:

`.claude/dbz-workflow.config.md`:

```markdown
# dbz-workflow 設定

## プロジェクト情報

**プロジェクト名**: claude-code-manual
**技術スタック**: Markdown, Claude Code
**テストコマンド**: npm run test
**E2Eコマンド**: E2Eテストなし

## 監査エージェント設定

| エージェント | 有効 | 備考 |
|-------------|------|------|
| audit-black-hacker | true | Webプロジェクトで有効 |
| audit-white-hacker | true | Webプロジェクトで有効 |
| audit-doc | true | 必須（全プロジェクト共通） |
| audit-i18n | false | 多言語対応プロジェクトのみ |
| audit-design | false | デザインシステム採用プロジェクトのみ |
| audit-a11y | true | Webフロントエンドプロジェクトのみ |
| audit-link | true | HTMLページを持つプロジェクトのみ |
| audit-test | true | テストスイートがあるプロジェクトのみ |

## モデル設定

<!-- 以下はカスタマイズ例です。プラグインのデフォルトから変更したいエージェントのみ記載してください。 -->
<!-- デフォルト: implementer=opus, reviewer=opus, plan-reviewer=opus, config-wizard=opus, audit-black-hacker=sonnet, audit-white-hacker=sonnet, 他の監査エージェント=sonnet -->

| エージェント | モデル | 備考 |
|-------------|--------|------|
| plan-reviewer | sonnet | コスト削減のため Sonnet に変更 |
| config-wizard | sonnet | コスト削減のため Sonnet に変更 |

指定可能なモデル: `sonnet`, `opus`, `haiku`

## カスタムルール

- コミットメッセージは日本語で記述
- PRタイトルにIssue番号を含める
```

**ペルソナ設定で「共通ペルソナ」または「エージェント別ペルソナ」を選択した場合**:

追加で `.claude/dbz-workflow/` ディレクトリに以下のファイルが生成されます：

- `reviewer.persona.md`
- `plan-reviewer.persona.md`
- `implementer.persona.md`
- `audit-doc.persona.md`

各ペルソナファイルの形式:

**作品名が入力された場合の例**:
```markdown
# reviewer ペルソナ設定

**名前**: ベジータ
**役割**: コードレビューの王子
**作品**: アニメ「ドラゴンボールZ」
**一人称**: オレ
**語尾**: 〜だ、〜だな、〜だろう
**口調の特徴**:
- 断定的で自信に満ちた口調
- 指摘時は「甘いな」「話にならん」などの表現
- 承認時は「悪くない」「認めてやろう」などの表現
```

**作品名が省略された場合の例**:
```markdown
# reviewer ペルソナ設定

**名前**: レビュー太郎
**役割**: コードレビュー担当
**一人称**: 私
**語尾**: 〜です、〜ます
**口調の特徴**:
- 丁寧で明確な表現
- 指摘時は理由と改善案をセットで提示
- 承認時は具体的な良い点を挙げる
```

> **注**: 作品名が省略された場合、`**作品**:` 行自体を出力しません。

## プロジェクトタイプ別の推奨設定

| プロジェクトタイプ | 推奨有効 | 推奨無効 |
|------------------|---------|---------|
| Webフロントエンド | black-hacker, white-hacker, doc, a11y, link, test | i18n, design |
| バックエンドAPI | black-hacker, white-hacker, doc, test | i18n, design, a11y, link |
| CLIツール | black-hacker, white-hacker, doc, test | i18n, design, a11y, link |
| ライブラリ | black-hacker, white-hacker, doc, test | i18n, design, a11y, link |
| ドキュメントのみ | doc, link | black-hacker, white-hacker, i18n, design, a11y, test |

**上書きルール**:
- デザインシステムを使用している場合: `audit-design` を有効化
- i18n対応している場合: `audit-i18n` を有効化

## 既存設定ファイルがある場合

既存の `.claude/dbz-workflow.config.md` が見つかった場合、上書き確認が表示されます：

```
既存の設定ファイルが見つかりました。

上書きしますか？
a) はい（既存設定を破棄して新規作成）
b) いいえ（処理を中断）
```

「いいえ」を選択した場合、処理は中断され、既存設定は保持されます。

---

## .claude/ ディレクトリが存在しない場合

自動的に `.claude/` ディレクトリを作成してから設定ファイルを生成します：

```bash
mkdir -p .claude
```

## カスタマイズ

生成された設定ファイルは手動で編集できます：

### ペルソナのカスタマイズ

**方法1: エージェント別ペルソナファイル（推奨）**

`.claude/dbz-workflow/{エージェント名}.persona.md` を編集：

```markdown
# reviewer ペルソナ設定

**名前**: ベジータ
**役割**: コードレビューの王子
**作品**: アニメ「ドラゴンボールZ」
**一人称**: オレ
**語尾**: 〜だ、〜だな、〜だろう
**口調の特徴**:
- 断定的で自信に満ちた口調
- 指摘時は「甘いな」「話にならん」などの表現
- 承認時は「悪くない」「認めてやろう」などの表現
```

**方法2: メイン設定ファイル（全エージェント共通）**

`.claude/dbz-workflow.config.md` に追加：

```markdown
## ペルソナ設定

**名前**: レビュー担当
**役割**: コードレビュー担当者
（以下省略）
```

### カスタムルールの追加

```markdown
## カスタムルール

- コミットメッセージは日本語で記述
- PRタイトルにIssue番号を含める
- 絵文字を使用しない
- 関数名はキャメルケース
```

---

## 次のステップ

設定ファイル生成後、以下のワークフローを開始できます：

### 実装計画の作成

```bash
/dbz-plan 123
```

Issue #123の実装計画を作成し、`plan-reviewer` でレビューします。

### PRワークフローの開始

```bash
/dbz-pr 123
```

Issue #123のPRワークフローを開始します（ブランチ作成 → 実装 → PR作成 → レビュー）。

---

## トラブルシューティング

### 設定ファイルが見つからない

```bash
ls -la .claude/dbz-workflow.config.md
```

ファイルが存在しない場合、`/dbz-init` を再実行してください。

### 設定が反映されない

1. `.claude/dbz-workflow.config.md` の内容を確認
2. フォーマットが正しいか確認（見出し、キー形式）
3. ワークフローを再実行
