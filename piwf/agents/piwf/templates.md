# PIWF テンプレート集

## 1. 申し送りリスト

Conditionalゲート通過時に必ず作成せよ。

```markdown
## 申し送りリスト（Phase [N] → Phase [N+1]）

| # | 指摘事項 | 指摘者 | ランク | 対応期限 | ステータス |
|---|---|---|---|---|---|
| 1 | ... | ... | ブロッカー/重要/将来 | Phase N中 / v1.1 | 完了/持ち越し |
```

## 2. 変更履歴サマリー

文書が改訂された場合、必ず冒頭に記載せよ。

```markdown
## 変更履歴サマリー（v[N-1] → v[N]）

| 項目 | 変更前 | 変更後 | 理由（指摘者） |
|---|---|---|---|
| ... | ... | ... | ...（C指摘 / G指摘 等） |
```

## 3. レビュー依頼

ゲートレビューまたは外部レビュアーへの依頼時に使用せよ。

```markdown
【レビュー依頼】
- 対象: [ファイル名] v[バージョン]
- フェーズ: Phase [N]
- 前回からの変更: [変更履歴サマリー参照]
- 特に見てほしい点: [具体的に]
- 前回の申し送り対応状況: [申し送りリスト参照]
- 期限: [日付]
```

## 4. 仮定レジストリ

設計フェーズで外部システムや技術的仮定を管理するために使用せよ。

```markdown
## 仮定レジストリ

| # | 仮定 | 検証可能 | 検証方法 | 検証状態 |
|---|------|----------|----------|----------|
| 1 | (例) OSV-Scannerのバイナリ名形式が `osv-scanner_linux_amd64` である | Yes | `gh api repos/google/osv-scanner/releases/tags/v2.0.0 --jq '.assets[].name'` | 未検証 |
| 2 | (例) GHAランナーにslsa-verifierが搭載されている | Yes | `runs-on: ubuntu-latest` で `which slsa-verifier` を確認 | 未検証 |
| 3 | (例) 将来のバージョンでも命名規則が維持される | No | -- | リスクとして記載 |
```

### 検証状態の値

| 値 | 意味 |
|---|------|
| 未検証 | まだ検証を実施していない |
| 検証済み | 一次ソースで事実を確認した |
| 検証失敗 | 検証の結果、仮定が誤りだった（設計修正が必要） |
| リスクとして記載 | 検証不可能な仮定。リスク対策を併記すること |

## 5. 実装ハンドオフ文書

Phase 3完了時に自動生成せよ。

```markdown
# [プロダクト名] 実装ハンドオフ文書
> 生成日: [日付]

## プロジェクト概要
[コンセプト定義から1-2文で記載]

## 成果物一覧
| Phase | ファイル | 概要 |
|---|---|---|
| 0 | piwf-output/{プロジェクト名}/phase0-validation.md | コンセプト定義、競合分析 |
| 1 | piwf-output/{プロジェクト名}/requirements-v{N}.md | 要件定義書 |
| 2 | piwf-output/{プロジェクト名}/legal-risk-analysis-v1.md | 法務リスク分析 |
| 2 | piwf-output/{プロジェクト名}/design-doc-v{N}.md | 設計メモ |
| 3 | piwf-output/{プロジェクト名}/api-design.md | API設計書 |
| 3 | piwf-output/{プロジェクト名}/error-handling.md | エラーハンドリング設計 |
| 3 | piwf-output/{プロジェクト名}/test-strategy.md | テスト戦略 |
| 3 | (※下記参照) | デザインシステム |
| 3 | piwf-output/{プロジェクト名}/mock-plan.md | モック計画（UI存在時） |
| 3 | piwf-output/{プロジェクト名}/library-notes.md | ライブラリ対策 |
| 3 | piwf-output/{プロジェクト名}/task-breakdown.md | タスク分割表 |
| - | CLAUDE.md（プロジェクトルート） | 実装時の行動指針 |

### デザインシステムのパスについて
- 既存の `design-system/{プロジェクト名}/MASTER.md`（ui-ux-pro-max 生成）がある場合: そのパスを記載する
- PIWF で生成した場合: `piwf-output/{プロジェクト名}/design-system/MASTER.md` を記載する

## 申し送り事項サマリー
[全Phaseの申し送りで未消化のものがあれば記載]

## 実装開始ガイド
1. CLAUDE.md を読む（実装時の行動指針）
2. タスク分割表を確認（作業順序）
3. 各タスクに対応する設計文書を参照
```
