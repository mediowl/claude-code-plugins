# Phase 0: バリデーション（アイデア検証）

## 目的
やる価値があるかを素早く判断する。

## ユーザーがPhase 0を開始したら

### Step 0: インプット確認
以下の必須情報が揃っているか確認せよ。不足があれば先に質問すること。

**必須（なければ質問）:**
1. 何を作りたいか（ざっくりでOK）
2. MVP期間（何週間で作りたいか）
3. 開発体制（個人/チーム、Claude Code使用の有無）

**あれば精度が上がる（なければスキップ）:**
4. ターゲットユーザーのイメージ
5. 技術スタックの希望や制約
6. 参考にしたいサービス
7. マネタイズの方針
8. 過去プロジェクトの振り返り（/piwf-retro スキルで生成した `piwf-retrospective.md`、2回目以降）

必須3項目が揃ったら Step 1 に進め。

### Step 1: コンセプト定義
ユーザーにプロダクトの概要をヒアリングし、以下を**1文ずつ**で定義せよ:
- **何を作るか**（プロダクト名＋一言説明）
- **誰のために**（ターゲットユーザー）
- **なぜ必要か**（解決する課題）

出力例:
```
コンセプト: 「TanPen」— 短編小説に特化した投稿プラットフォーム
ターゲット: 趣味で短編を書くライト層＋通勤中にサクッと読みたい人
課題: 既存サービスは長編中心でUIが短編に最適化されていない
```

**プロダクト名のケバブケース化**: コンセプト定義で確定したプロダクト名をケバブケースに変換し、以後の成果物出力先ディレクトリ名として使用せよ（例: `TanPen` → `tan-pen`）。

### Step 2: 競合分析
ユーザーから情報を聞き出すか、Web検索で調査し、以下の表を埋めよ:

```markdown
| 競合サービス | 強み | 弱み | 本プロダクトの差別化 |
|---|---|---|---|
| ... | ... | ... | ... |
```

### Step 3: 技術フィージビリティ概算
以下を確認せよ:
- 想定する技術スタック
- MVP期間の見積もり
- 明らかな技術的ブロッカーの有無

### Step 4: 成果物の出力
上記の結果をまとめたMarkdownファイルを作成せよ。ファイル名: `piwf-output/{プロジェクト名}/phase0-validation.md`

## 完了条件
以下がすべて満たされていることを確認せよ:
- [ ] コンセプトが1文で説明できている
- [ ] 明確な差別化ポイントがある
- [ ] 技術的に実現可能と判断できる
- [ ] 成果物ファイルが `piwf-output/{プロジェクト名}/` に出力されている

## 次のアクション
Phase 0 の成果物が完了した。ゲートレビューを自動実行する。
→ gate-review.md を読み込み、Phase 0 のゲートレビューを実施せよ。
→ Pass の場合: Phase 1 の開始を案内せよ。同一セッションで続行する場合は Phase 1 を開始せよ。
→ Conditional の場合: 申し送りリストを作成し、Phase 1 の開始を案内せよ。
→ Fail の場合: ブロッカー指摘をユーザーに提示し、解決後に再度ゲートレビューを実施せよ。
