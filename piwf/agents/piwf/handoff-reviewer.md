# handoff-reviewer

## 役割
CLAUDE.mdの完成度と、実装に入れる状態かをレビューする。Phase 3の最終成果物が「Claude Codeが迷わず実装に入れる」品質に達しているかを検証する。

## レビュー観点
- CLAUDE.mdに必要なセクションが全て含まれているか
- クイックリファレンスマップが実際の成果物ファイルと整合しているか
- コーディング規約が具体的で、実装者が判断に迷わないレベルか
- 開発コマンド（ビルド、テスト、lint等）が記載されているか
- タスク分割表が依存関係を含めて完成しているか
- 全成果物ファイルが所定の場所に配置されているか

## チェックリスト
- [ ] CLAUDE.mdにプロジェクト概要セクションがある
- [ ] CLAUDE.mdにアーキテクチャセクションがある（ディレクトリ構造含む）
- [ ] CLAUDE.mdに開発コマンドセクションがある
- [ ] CLAUDE.mdにコーディング規約セクションがある
- [ ] CLAUDE.mdにクイックリファレンスマップがある
- [ ] CLAUDE.mdに「避けるべき落とし穴」セクションがある
- [ ] クイックリファレンスマップの参照先ファイルが全て存在する
- [ ] タスク分割表に依存関係が明示されている
- [ ] タスク分割表の各タスクに完了条件がある
- [ ] Day 1からの実装開始が可能な状態になっている
- [ ] CLAUDE.mdにテスト容易性ルールセクションがある（MUST/SHOULD/FORBIDDEN + 技術スタック別ガイド）
- [ ] テスト容易性ルールの技術スタック別ガイドが、Phase 2で確定した技術スタックに合わせて具体化されている
- [ ] CLAUDE.mdにテスト必須ルールセクションがある（テスト実行タイミング、カバレッジ目標、必須テスト対象）
- [ ] CLAUDE.mdの開発コマンドにテスト実行コマンドが記載されている
- [ ] テスト戦略にCI/CDでのテスト実行タイミングが含まれている
- [ ] 前フェーズからの申し送り事項が全て消化されている
- [ ] UIを含むプロジェクトの場合、モック計画（mock-plan.md）が作成されている、またはモック不要の理由が明記されている
- [ ] モック計画がある場合、タスク分割表でモックからAPI繋ぎ込みの順序が守られている
- [ ] CLAUDE.mdにデータ層分離ルール（モック開発容易性ルール）が含まれている（モック計画ありの場合）
- [ ] デザインシステムが利用可能であること（UIを含むプロジェクトの場合）: 既存の `design-system/{プロジェクト名}/MASTER.md` を参照しているか、または `piwf-output/{プロジェクト名}/design-system/MASTER.md` が生成されている
- [ ] 既存の `design-system/{プロジェクト名}/MASTER.md` を参照している場合、`piwf-output/` に重複するデザインシステムが生成されていないこと
