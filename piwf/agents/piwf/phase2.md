# Phase 2: 設計・法務レビュー

## 目的
技術設計を固め、法務リスクを洗い出す。

## ユーザーがPhase 2を開始したら

### Step 0: 前提確認
1. Phase 1の成果物（`piwf-output/{プロジェクト名}/requirements-v1.md`）を Read ツールで読み込め
2. 申し送りリストがあれば冒頭で消化状況を確認せよ

### Step 1: アーキテクチャ設計
要件定義書をもとに以下を確定せよ:
- ディレクトリ構造
- レイヤー構成（プレゼンテーション/ビジネスロジック/データアクセス）
- 各レイヤーの責任範囲

### Step 2: 仮定の洗い出しと検証

設計に含まれる外部システムへの依存や技術的仮定を洗い出し、検証する。仮定レジストリの書式は `templates.md` を参照。

#### Step 2a: 分類

設計文書から以下のカテゴリで仮定を抽出し、仮定レジストリに記録せよ:
- **外部APIの仕様**: エンドポイント、レスポンス形式、命名規則等
- **外部ツール・バイナリ**: ダウンロードURL、ファイル名形式、動作要件等
- **実行環境**: ランナー、コンテナ、OS に搭載されるツール・ライブラリ等
- **サードパーティサービス**: 料金体系、レート制限、SLA等

各仮定について「検証可能（Yes）」か「検証不可能（No）」かを判定せよ。

#### Step 2b: 実地検証（Spike）

検証可能な仮定について、ワークフロー内で実行可能な方法で検証せよ:
- CLI コマンドで一次ソースを確認（例: `gh api` でリリースアセット名を取得）
- 公式ドキュメントの該当ページを WebFetch で取得し確認
- 検証結果を仮定レジストリに反映し、検証状態を「検証済み」または「検証失敗」に更新せよ

**検証失敗の場合**: 仮定が誤りであることが判明したら、設計文書を修正せよ。

**制限事項**: Spike は「ワークフロー内で実行可能な検証のみ」とする。外部環境へのデプロイや課金が発生する操作は行わない。

#### Step 2c: 検証不可能な仮定の扱い

検証不可能な仮定（将来の互換性、サービスの可用性等）については:
- 仮定レジストリの検証状態を「リスクとして記載」に設定せよ
- 仮定が崩れた場合の影響範囲と対策（フォールバック手段等）を設計文書に記載せよ

**外部依存がない設計の場合**: 仮定レジストリに「外部依存なし」と明記し、本ステップをスキップしてよい。

### Step 3: DB設計の詳細化
- RLS（Row Level Security）ポリシーの定義
- DB制約（ユニーク制約、CHECK制約、トリガー）
- インデックス設計
- マイグレーション方針

### Step 4: セキュリティ設計
以下の方針を確定せよ:
- 認証・認可の仕組み
- XSS/CSRF対策
- キャッシュ制御（ユーザー依存ページはキャッシュ禁止等）
- エラー時の情報開示方針（ここでは方針のみ。実装ルールはPhase 3で確定）

### Step 5: 法務リスク分析
プロダクトの性質に応じて以下を分析せよ:
- 利用規約で定めるべき事項
- プライバシーポリシーの必要項目
- 通報・コンテンツモデレーションの要否
- 外部サービス利用に伴う法的義務（越境移転等）

法務分析の結果は別ファイルとして出力: `piwf-output/{プロジェクト名}/legal-risk-analysis-v1.md`

法務分析書の末尾に以下の免責文を必ず含めること:
```markdown
---
**免責事項**: 本文書はAIによる自動分析の結果であり、法的助言を構成するものではありません。実際の法務判断については、必ず資格を持つ弁護士または法務専門家にご相談ください。
```

### Step 6: 変更履歴サマリーの作成
Phase 1の要件定義書がレビューを経て改訂された場合、変更履歴サマリーを冒頭に記載せよ。
書式は `templates.md` を参照。

### イテレーション管理
このフェーズ内で「ドラフト→レビュー→改訂」を繰り返す場合:
- **最大3イテレーション**を目安とせよ
- 超える場合はユーザーにスコープ見直しを提案せよ

### 成果物の出力
以下のファイルを出力せよ:
- `piwf-output/{プロジェクト名}/requirements-v{N}.md`（改訂版の要件定義書）
- `piwf-output/{プロジェクト名}/legal-risk-analysis-v1.md`（法務リスク分析書）
- `piwf-output/{プロジェクト名}/design-doc-v1.md`（設計メモ）

## 完了条件
- [ ] 技術設計が実装可能なレベルに具体化されている
- [ ] 仮定レジストリが作成されている（外部依存なしの場合はその旨を明記）
- [ ] 検証可能な仮定がすべて検証済みである
- [ ] 検証不可能な仮定にリスク対策が記載されている
- [ ] DB設計にRLS・制約が定義されている
- [ ] セキュリティ方針が確定している
- [ ] 法務リスクが洗い出され対策が明記されている
- [ ] 改訂文書に変更履歴サマリーが付いている
- [ ] 全成果物ファイルが `piwf-output/{プロジェクト名}/` に出力されている
- [ ] 申し送り事項が消化済み

## 次のアクション
Phase 2 の成果物が完了した。ゲートレビューを自動実行する。
→ gate-review.md を読み込み、Phase 2 のゲートレビューを実施せよ。
→ Pass の場合: Phase 3 の開始を案内せよ。同一セッションで続行する場合は Phase 3 を開始せよ。
→ Conditional の場合: 申し送りリストを作成し、Phase 3 の開始を案内せよ。
→ Fail の場合: ブロッカー指摘をユーザーに提示し、解決後に再度ゲートレビューを実施せよ。
