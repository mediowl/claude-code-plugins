# Phase 3: 開発準備

## 目的
実装に入れる状態にする。Phase 1-2の「方針」を「実装に渡せるレベルの仕様」に具体化する。

## ユーザーがPhase 3を開始したら

### Step 0: 前提確認
1. Phase 2の成果物（`piwf-output/{プロジェクト名}/` 配下の要件定義書、法務分析書、設計メモ）が提供されているか確認せよ
2. 申し送りリストがあれば冒頭で消化状況を確認せよ

### Step 1: 設計の詳細化

以下の設計ドキュメントを作成せよ。Phase 1-2との境界に注意すること。

#### API設計書
- エンドポイント一覧、命名規則
- リクエスト/レスポンスの統一フォーマット
- エラーレスポンスの統一フォーマット
- ※Phase 2では「何のAPIが必要か」を決定済み。ここでは**実装仕様を確定**する

#### エラーハンドリング設計
- エラー分類（バリデーション/認証/権限/システム）
- 各分類のログレベル（info/warn/error）
- ユーザーに表示するメッセージの方針
- ※Phase 2のセキュリティ設計では「エラー時にどこまで情報を出すか」の方針を決定済み。ここでは**実装ルールを確定**する

#### テスト戦略
Phase 1で定義したテスト要件（テスト必須有無、CI自動実行フェーズ、カバレッジ最小目標）を具体化せよ:
- 単体/統合/E2Eの役割分担
- カバレッジ目標（Phase 1で定義した最小目標を基に具体的な数値を確定）
- テストファイルの配置ルール
- モック方針
- CI/CDでのテスト実行:
  - テスト実行タイミング（例: PR作成時、マージ前）
  - テスト実行コマンド（CI環境向け）
  - 失敗時のブロック条件（例: テスト失敗でマージをブロック）

#### デザインシステム文書

##### 既存デザインシステムの検出（必須）
デザインシステム文書の作成前に、以下の順序で既存ファイルの存在を確認せよ:

1. `design-system/{プロジェクト名}/MASTER.md` の存在を確認する
2. `design-system/` ディレクトリ自体の存在を確認する

##### 分岐ロジック

**ケースA: 既存の MASTER.md がある場合**（`design-system/{プロジェクト名}/MASTER.md` が存在する）
- `piwf-output/` にデザインシステムを**生成しない**
- 既存の `design-system/{プロジェクト名}/MASTER.md` をそのまま参照する
- handoff.md の成果物一覧に既存パス `design-system/{プロジェクト名}/MASTER.md`（ui-ux-pro-max 生成）を記載する
- Step 5 の成果物配置で `piwf-output/{プロジェクト名}/design-system/MASTER.md` を作成しない

**ケースB: design-system/ ディレクトリ自体が存在しない場合**（ui-ux-pro-max 未実行）
- ユーザーに以下のメッセージを表示し、確認を取る:
  > `design-system/` ディレクトリが見つかりません。ui-ux-pro-max プラグインでデザインシステムを生成することを推奨します。ui-ux-pro-max を先に実行しますか？それとも PIWF でデザインシステムを生成しますか？
- ユーザーが ui-ux-pro-max の実行を選択した場合: PIWF を一時中断し、実行後に再開する
- ユーザーが PIWF での生成を選択した場合: ケースC と同じ手順で生成する

**ケースC: design-system/ は存在するが MASTER.md がない場合**
- 通常の PIWF 生成フローで `piwf-output/{プロジェクト名}/design-system/MASTER.md` を生成する

##### デザインシステムの要件（ケースB/C で PIWF 生成する場合）
出力先: `piwf-output/{プロジェクト名}/design-system/MASTER.md`

以下の項目を含めること（ツール非依存の要件）:
- カラーパレットの具体値（HEXコード）
- タイポグラフィ（フォント、サイズ）
- スペーシングルール
- コンポーネント規約
- ※Phase 1で「方向性」を決定済み。ここでは**実装に使える具体値を確定**する

#### モック計画（条件付き）
> UIを含むプロジェクトの場合に作成する。UIなしプロジェクトはスキップ可（スキップ時はその理由を明記すること）。

出力先: `piwf-output/{プロジェクト名}/mock-plan.md`

以下の項目を含めること:
- モック対象ページ一覧
- ダミーデータ方針
- データ層分離ルール
- モックレビュー方針
- モックから本番への移行方針

### Step 2: ライブラリ対策ドキュメント
使用する主要ライブラリについて、以下を事前調査して記録せよ:
- 既知のハマりどころ
- 推奨される使い方
- やってはいけないパターン
- バージョン固有の注意点

Deep ResearchやWeb検索を活用して調査すること。

### Step 3: タスク分割表の作成
以下のルールでタスクを分割せよ:
- 1タスク = 1機能 or 1ページ
- 依存関係を明示（DB → API → UI の順）
- 各タスクの完了条件を明記
- Day 1〜N の割り当て

#### モック→API順序制約（UIを含むプロジェクトかつモック計画がある場合）
- モック作成タスクをAPI実装タスクの前に配置せよ
- モック作成タスクの完了条件に「ブラウザ確認済み」を含めよ
- モックレビュー完了をAPI繋ぎ込みタスクの前提条件とせよ

## テスト容易性ルール

> テストを書きやすく保守しやすいコードにするためのルール。具体的なテスト戦略（カバレッジ目標・配置ルール等）は test-strategy.md を参照。

### MUST
- 外部依存（DB・API・ファイルシステム等）はインターフェースまたは関数引数で注入可能にせよ
- 1関数/1メソッドの責務を単一にせよ（テスト対象が明確になる）
- 副作用を持つ処理と純粋なロジックを分離せよ

### SHOULD
- テスト用のファクトリ関数やビルダーを用意し、テストデータの作成を容易にする
- 公開APIの境界でテストし、内部実装への依存を避ける
- 環境依存値（URL・タイムアウト値等）は設定として外部化する

### FORBIDDEN
- テスト不可能なグローバル状態への依存（シングルトンの直接参照等）
- テスト内での本番外部サービスの直接呼び出し
- 時刻・乱数などの非決定的な値のハードコード

### 技術スタック別の実装ガイド
※ Phase 2 で確定した技術スタックに応じて、以下のように具体化すること:
- DI の実装方法（例: コンストラクタ注入 / React Context / 関数引数 / etc.）
- モック戦略（例: jest.mock / MSW / テスト用Repository実装 / etc.）
- テストヘルパーの配置先（例: src/__tests__/helpers/）

> - 「テスト必須ルール」: CI運用方針（いつ、何を実行するか）

## デザイン変更容易性ルール

> 色・余白・タイポグラフィ等のデザイン変更時に修正箇所を最小化するためのルール。具体的なカラー値・フォント・スペーシングは design-system/MASTER.md を参照。

### MUST
- デザイントークン（色/余白/角丸/影/タイポグラフィ）を一元管理せよ
- 共通UIコンポーネント層（Button, Input, Card 等）を設け、variant/size で差異を吸収せよ
- ページコンポーネントで直接スタイルを定義するな（コンポーネント経由で表現せよ）

### SHOULD
- ページテンプレート層（List/Detail/Form 等）を設け、レイアウトの一貫性を確保する
- コンポーネントの見た目の差異は variant として明示的に定義し、例外的なスタイル上書きを避ける

### FORBIDDEN
- ハードコードされた色値やマジックナンバーのスペーシングをページ内に直書き
- 同種UIの重複実装（既存コンポーネントを拡張せよ）
- トークン定義を経由しない任意値の多用

### 技術スタック別の実装ガイド
※ Phase 2 で確定した技術スタックに応じて、以下のように具体化すること:
- トークン管理の実装方法（例: Tailwind theme + CSS variables / CSS Modules + design-tokens.ts / etc.）
- コンポーネント配置先（例: src/components/ui/）
- 禁止パターンの具体例（例: arbitrary values `px-[13px]` の禁止）

> UIを含むプロジェクトかつモック計画がある場合、以下のセクションも含めること。

## モック開発容易性ルール

> モック段階からAPI繋ぎ込みまでスムーズに移行するためのルール。具体的なモック計画は mock-plan.md を参照。

### MUST
- データ取得は専用関数/フックに分離せよ（UI層で直接fetchしない）
- ダミーデータは別ファイルに配置せよ（コンポーネント内にハードコードしない）
- API差し替え時にUIコンポーネントの修正が不要な設計にせよ

### SHOULD
- ローディング/エラー/空状態のUIをモック段階で作成する
- データ取得関数のインターフェースをAPI設計書と揃える

### FORBIDDEN
- UIコンポーネント内でのダミーデータ直書き
- データ取得ロジックとUI表示ロジックの混在

## 避けるべき落とし穴
[過去プロジェクトの振り返りから引き継いだハマりポイント]
[初回プロジェクトでは「今後のプロジェクトで蓄積していく」と記載]

### Step 5: 成果物の配置
以下のファイルを配置せよ。CLAUDE.md以外は全て `piwf-output/{プロジェクト名}/` 配下に統一する:
- `CLAUDE.md`（プロジェクトルート — 唯一の例外）
- `piwf-output/{プロジェクト名}/api-design.md`（API設計書）
- `piwf-output/{プロジェクト名}/error-handling.md`（エラーハンドリング設計）
- `piwf-output/{プロジェクト名}/test-strategy.md`（テスト戦略）
- `piwf-output/{プロジェクト名}/design-system/MASTER.md`（デザインシステム — ケースC の場合のみ。ケースA では既存パスを参照）
- `piwf-output/{プロジェクト名}/library-notes.md`（ライブラリ対策）
- `piwf-output/{プロジェクト名}/task-breakdown.md`（タスク分割表）
- `piwf-output/{プロジェクト名}/mock-plan.md`（モック計画 — UIを含むプロジェクトの場合）

### Step 6: ハンドオフ文書の生成
成果物の配置が完了したら、実装ハンドオフ文書を自動生成せよ。
- ファイル名: `piwf-output/{プロジェクト名}/handoff.md`
- 書式: `templates.md` のテンプレート5「実装ハンドオフ文書」に従う
- 責務: 成果物の目次・全体マップとして機能する（CLAUDE.mdは「実装時の行動指針」）

## 次のアクション
Phase 3 の成果物が完了した。ゲートレビューを自動実行する。
→ gate-review.md を読み込み、Phase 3 のゲートレビューを実施せよ。
→ Pass の場合: PIWFスコープ完了を宣言し、dbz-workflow等の別ワークフローでの実装開始を案内せよ。
→ Conditional の場合: 申し送りリスト（templates.md 書式）を作成し、ユーザーに提示した上でPIWFスコープ完了を宣言せよ。
→ Fail の場合: ブロッカー指摘をユーザーに提示し、解決後に再度ゲートレビューを実施せよ。
