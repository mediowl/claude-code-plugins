# Phase 1: 要件定義

## 目的
何を作るかを具体化し、要件定義書を完成させる。

## ユーザーがPhase 1を開始したら

### Step 0: 前提確認
1. Phase 0の成果物（`piwf-output/{プロジェクト名}/phase0-validation.md`）を Read ツールで読み込め
2. Phase 0からの申し送りリストがあれば、冒頭で消化状況を確認せよ

### Step 1: 機能要件の洗い出し
ユーザーにヒアリングしながら、全機能をMust / Should / Couldに分類せよ:

```markdown
| 機能 | 詳細 | 優先度 |
|---|---|---|
| ... | ... | Must / Should / Could |
```

**重要**: Must機能がMVP期間内に収まるかを常に意識せよ。収まらない場合はユーザーにスコープ削減を提案すること。

### Step 2: 非機能要件の定義
以下を確認・定義せよ:
- パフォーマンス目標（ページ読み込み時間等）
- セキュリティ要件（認証方式、データ保護）
- 可用性目標
- デザインの方向性（カラー、トーン）
- テスト要件（以下を必ず確認すること）:
  - **テスト必須有無**: 自動テストを実施するか（基本は必須。純粋なドキュメントプロジェクト等、テスト不要な合理的理由がある場合のみ「不要」とする）
  - **CI自動実行フェーズ**: テストをCIで自動実行するタイミング（例: PR作成時、マージ前）
  - **カバレッジ最小目標**: 達成すべき最低限のカバレッジ水準（例: 80%）

### Step 3: データモデル設計
主要なエンティティとリレーションを定義せよ:
- テーブル一覧と主要カラム
- リレーション（1:N、N:N等）
- 命名規則

### Step 4: 技術スタック選定
以下を確定せよ:
- フロントエンド（フレームワーク、UIライブラリ）
- バックエンド（言語、フレームワーク）
- データベース
- ホスティング
- 認証方式

### Step 5: MVP期間との整合性チェック
Must機能×技術スタックで、MVP期間内に実装可能かを検証せよ。
不可能な場合、以下の選択肢をユーザーに提示:
- Must機能を削減する
- MVP期間を延長する
- 技術スタックを簡素化する

### Step 6: 成果物の出力
要件定義書をMarkdownファイルとして出力せよ。ファイル名: `piwf-output/{プロジェクト名}/requirements-v1.md`

必ず冒頭に以下を含めること:
```markdown
# [プロダクト名] 要件定義書 v1
> MVP開発期間: [N]週間
> 最終更新: [日付]
```

## 完了条件
- [ ] Must機能がMVP期間に収まる
- [ ] データモデルが定義されている
- [ ] 技術スタックが確定している
- [ ] テスト要件が定義されている（テスト必須有無、CI自動実行フェーズ、カバレッジ最小目標）
- [ ] 申し送り事項が消化済み
- [ ] 要件定義書ファイルが `piwf-output/{プロジェクト名}/` に出力されている

## 次のアクション
Phase 1 の成果物が完了した。ゲートレビューを自動実行する。
→ gate-review.md を読み込み、Phase 1 のゲートレビューを実施せよ。
→ Pass の場合: Phase 2 の開始を案内せよ。同一セッションで続行する場合は Phase 2 を開始せよ。
→ Conditional の場合: 申し送りリストを作成し、Phase 2 の開始を案内せよ。
→ Fail の場合: ブロッカー指摘をユーザーに提示し、解決後に再度ゲートレビューを実施せよ。
